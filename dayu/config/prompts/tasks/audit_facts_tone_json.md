审计目标：
- 审计“章节正文”是否存在事实/证据问题、结构/越界问题，以及少数会明显伤害买方报告质量的风格问题。
- 仅输出审计结果 JSON，不要改写正文，不要输出其它文字。

输出要求：
- 你只能输出 JSON 格式审计结果，禁止输出解释性文字。
- 输出必须是严格可解析 JSON。
- 第一字符必须是 { 或 [。
- 最后一个非空白字符必须是 } 或 ]。
- JSON 之外不得有任何字符。
- 输出 JSON schema：
{
  "pass": true/false,
  "class": "ok|evidence_insufficient|content_violation|style_violation",
  "violations": [
    {
      "rule": "E1|E2|E3|C1|C2|S1|S2|S3|S4|S5|S6|S7",
      "severity": "high|medium|low",
      "excerpt": "从本章原文复制的触发片段",
      "reason": "为什么违反/高风险（按协议口径解释）",
      "rewrite_hint": "如何改成合规表述（动作建议）"
    }
  ],
  "notes": ["可选：整体风险提示"]
}
- class 说明：
  - evidence_insufficient：出现任意 E 类违规且 pass=false
  - content_violation：出现阻断性的 C 类违规且 pass=false
  - style_violation：仅出现 S 类违规且 pass=false
  - ok：pass=true

审计原则（强制）：
- 优先发现会伤害报告质量的高价值问题，不要为低价值问题牺牲更准确的买方表述。
- 下方“章节结构”用于核对固定结构；“章节合同”用于核对本章必须回答什么、不得展开什么；“条件写作规则”用于判断条件可见的小节或条目是否允许出现；“审计规则范围”用于声明本轮要执行的规则集合。
- 若“当前审计模式”是“初始整章审计”，先列全问题，再统一给 `pass/class`；不要因为先发现一条高严重度问题就停止全文检查。
- 若“当前审计模式”是“修复后局部复审”，优先只复查“修复合同”命中的问题类型、相关 excerpt/slot 及其所在 bullet / 段落 / 小节的局部上下文；不要把未触及且与当前修复无关的整章内容重新按初始整章审计标准全文复扫。
- 若同一片段可能命中多条规则，先做同源归因，再给一个统一的 `rewrite_hint`；不得给出彼此冲突的修复建议。
- `rewrite_hint` 先修实质问题，再修样式问题；优先给出最短且可直接执行的动作。
- 对 `E1/E2`，若当前输入中的同源证据已足以稳定推出正确数值、正确口径或正确表述，`rewrite_hint` 应优先写“修正为……”；只有在无法稳定确定替代值时，才建议删除、降级或补证。
- 对不属于本章最低必需判断链的 unsupported claim，默认优先建议删除或降级，不要默认建议补证据。
- 若正文句子同时包含可保留的客观事实与不应保留的解释性拔高，不要整句一刀切打回；应优先指出需要删除、弱化或拆分的具体片段。
- 若某条内容在当前输入中被明确表达为后续跟踪对象、待验证问题、敏感性变量或需持续观察的信号，而非当前已成立事实、当前状态判断或具体数值，默认不触发 `E1/E2`。
- 当 evidence line 已指向能直接承载该断言的具体章节、注释或标题路径时，不得仅因为证据行未重复正文中的数字本身，就判定 `E1/E2`。
- `S7` 只用于提示“证据与出处”的可读性或格式可改进之处；只要来源与定位稳定可追溯，不应仅因不是某一种理想标题样式就报违约。

判定步骤（严格按顺序执行）：
1. 若“当前审计模式”是“初始整章审计”，先做全文扫描，按“审计口径”逐条检查，列出所有真实 `violations`。
2. 若“当前审计模式”是“初始整章审计”，输出 `violations` 前，必须对全文再做一次遗漏复扫，尤其复扫所有包含关键数字、比例、唯一性、份额、排名、区域拆分、客户集中度、收入拆分的句子，确认没有漏项，尽量一次列全真实问题。
3. 若“当前审计模式”是“修复后局部复审”，只围绕“修复合同”命中的 excerpt / slot 及其直接相邻上下文做局部复审；若本轮 patch 在局部上下文中引入新问题，可一并报告，但不要重新启动整章全文复扫。
4. 在当前审计模式下，`violations` 列全之后，再统一计算 `pass` 与 `class`：
   - 只要出现任意 `E1/E2/E3` → `pass=false`，`class="evidence_insufficient"`。
   - 否则，只要出现任意 `C1` → `pass=false`，`class="content_violation"`。
   - 否则，若中高优先级写作风格问题累计达到 2 条及以上 → `pass=false`，`class="style_violation"`。
   - 否则 → `pass=true`，`class="ok"`。
- 无论 pass 结果如何，所有 violations 均保留在输出中供后续修正。

审计口径（逐条检查）：
- 证据充分性：
  - E1) 正文存在关键断言但无可对应的证据条目；触发时按 high severity 处理。
  - E2) 正文出现不带证据的具体数字或口径；触发时按 high severity 处理。
  - E3) 证据条目指向的来源本身不可追溯或不可定位；包括 document id / accession / filing / URL / 标题路径不存在、无法落到原文，或明显不属于当前 ticker / 当前公司；触发时按 high severity 处理。对网页资料，若证据条目已提供页面标题、发布/访问日期和可追溯 URL，则默认不触发 `E3`；若只是缺少更细定位但仍可追溯，改为触发 `S7`。

- 内容合规性：
  - C1) 大量占位符/"未披露"/泛化描述导致信息无法闭环；触发时按 high severity 处理。
  - C2) 章节越界：正文在实质篇幅内展开了`章节合同`中明确禁止展开的话题；仅用于回答本章主问题的客观角色、技术或生态定位不算越界；触发时按 low severity 处理。

- 写作风格：
  - S1) 中文；买方报告口径；简洁、可读；不写情绪化或安抚性措辞；触发时按 medium severity 处理。
  - S2) 不做大段摘抄或逐句翻译；触发时按 medium severity 处理。
  - S3) 正文重复、机械或无必要地使用“公司披露/根据/报告称/原文指出”等引用原文套话；不适用于“证据与出处”。若某个引用前缀仅出现一次，且有助于区分事实、公司原文或公司预期，不触发 `S3`；触发时按 medium severity 处理。
  - S4) 将推断、归因或解释性判断写成硬事实，导致读者无法区分“事实”与“分析”；若句子本身已经明显是判断句，或位于“结论要点 / 当前判断 / 最关键的有利点或硬伤”等天然判断槽位，默认不触发；触发时按 low severity 处理。
  - S5) 将未来趋势、预测或情景判断写成当前事实，且未用“预计 / 预期 / 可能 / 到某年 / 指引”等自然语言做前瞻区分；若句子本身已自然表达其前瞻属性，默认不触发；触发时按 low severity 处理。
  - S6) 引用公司原文中的前瞻性表述时，未明确这是公司披露、管理层预期或公司指引，导致读者可能误以为是当前事实；若正文已用自然归因表达，默认不触发；触发时按 low severity 处理。
  - S7) "证据与出处"格式可改进，但来源与定位仍可追溯；触发时按 low severity 处理。
